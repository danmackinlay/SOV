# A new model changed our maths — the first SOV update

A word on what you're reading, since it's the first. SOV is a small
experiment: a handful of people in Australia working out whether a tiny
group could own and run its own large language model — the kind of
system you'd normally rent from a big company — instead of paying a
vendor forever. Before anyone buys hardware, we're renting the same
setup on cloud machines, to live with it and decide if it's worth it.
The reasoning is written up in two posts: [the
case](https://danmackinlay.name/notebook/aus_sovereign_llm.html) and
its [technical
companion](https://danmackinlay.name/notebook/aus_sovereign_llm_technical.html).
This covers roughly the first month, and one week of it changed our
minds.

## The question the whole project rests on

If a small collective bought one powerful computer, could it serve
enough people at once to be worth it? Fitting a model on a machine is
the easy part; the hard part is how many people can use it at the same
time before it slows to a crawl. The original write-up's numbers looked
fine for short exchanges. They were much less convincing for the work
we actually care about — long, back-and-forth sessions where an
assistant grinds through a big task. That use case justifies the whole
effort, and it was also the one our own arithmetic had us most worried
about.

## What moved

**The long-conversation problem was real, and then a new model mostly
dissolved it.** While a model holds a conversation it keeps a running
memory of everything said so far (the "KV cache"), in the same scarce,
fast space as the model itself. The longer the session, the more space
one person eats. Our numbers said a single power user on a long session
could fill the whole machine — an expensive single-user computer, not a
cooperative. It was the weakest point in the case, so we wrote it up as
[ADR
0007](https://github.com/danmackinlay/SOV/blob/main/docs/decisions/0007-deepseek-v4-and-the-concurrency-consequence.md)
rather than around it.

Then DeepSeek V4 landed in April 2026 — specifically the smaller
**V4-Flash** (284B-parameter, 13B-active) variant — using **sparse
attention** (DeepSeek Sparse Attention, DSA), which compresses the
conversation before storing it. A full million-token session costs
about 9.6 GB of that fast memory instead of tens of GB: roughly a 90%
cut (DeepSeek's published figure; we redid our arithmetic with it and
showed the working). The wall didn't vanish, but it moved out about
tenfold. The question changed from "we run out of room after one heavy
user" to "how much can the box actually push?" — measurable and
fundable rather than a dealbreaker, and now load-bearing for the
economic case. One caveat: these are still pencil-and-paper estimates
on published figures, not our own measurements. Replacing them with
real benchmarks is on the list.

**We added a smaller second track: a model on a laptop.** The audition
is deliberately cloud-based, which left an everyday gap — offline work,
or daily use without renting a GPU. So we set up a parallel [Apple
laptop
track](https://github.com/danmackinlay/SOV/blob/main/docs/decisions/0004-apple-laptop-personal-track.md)
running smaller models locally (via Apple's
[`mlx-lm`](https://github.com/ml-explore/mlx-lm)) but deliberately
mirroring the main experiment's plumbing. A laptop doesn't replace the
cooperative; it answers a different question, and daily use of it is
unglamorous practice for the real thing.

**We changed our minds about the chat interface.** The browser chat we
first picked ([Open WebUI](https://github.com/open-webui/open-webui))
relicensed at v0.6.6 (April 2025) from BSD-3-Clause to a custom,
non-open licence (the Open WebUI License, with a contributor agreement
and a >50-user branding clause). Since not trading one kind of lock-in
for another is the entire point of this project, we switched now to a
permissively licensed alternative
([LibreChat](https://github.com/danny-avila/LibreChat), MIT) rather
than carry the tension and swap later. Small, but exactly the kind of
trade-off this project exists to surface ([ADR
0003](https://github.com/danmackinlay/SOV/blob/main/docs/decisions/0003-audition-surface-and-auth.md)).

## What we still don't know

Fitting more conversations in memory is not the same as serving them
*fast*: the bottleneck may just move somewhere we haven't measured.
There are no real throughput numbers on real hardware yet — everything
above is arithmetic until the benchmarking phase replaces it. And the
"de-censored" (abliterated) model piece carries a fresh unknown: that
tooling is more mature on older models than on this one, so we can't
assume it transfers.

That's the first month. Thanks for following along.
